# Changelog

All notable changes to this project should be documented in this file.

The format is intentionally human-curated with an `Unreleased` section that collects work before release.

## Unreleased

### Fixed
- **Mesh Topography's on-canvas pitch handle can now tip the model below the horizon — the gizmo pitch range matches 3D Spiral (v1.1.118).** Following the v1.1.117 handle-wiring fix, the on-canvas rotation gizmo still clamped pitch to `0…89°`, so you could never drag the model to look up at the underside even though the Pitch *slider* spans `−180…180°`. The gizmo's `pitchMin`/`pitchMax` widen from `0…89` to `−90…90`, matching the working `spiral3d` entry, so the canvas handle now tips a full quarter-turn each way. Slider range and defaults are unchanged. (RGR: `tests/unit/renderer-3d-rotation-control.test.js` — dragging the mesh-topography pitch handle past the horizon now reaches `−90` where the old spec clamped it to `0`. Full unit suite green.)
- **Mesh Topography's on-canvas rotation handles now actually rotate the model, and the missing Roll handle is back — three handles, all live (v1.1.117).** The v1.1.113 param rename (`rotate → yaw`, `tilt → pitch`, plus the new `roll`) updated the slider controls and the generator, but missed the renderer's `ROTATION_3D_SPECS.meshTopography` entry that drives the *on-canvas* rotation gizmo. It still declared `yawParam:'rotate'` / `pitchParam:'tilt'` with no `rollParam`, so dragging the canvas handles wrote to the now-dead `rotate`/`tilt` keys (the generator reads only `yaw`/`pitch`/`roll`, so the graphic never moved) and the roll handle was never created (only two handles drew instead of three). The spec now matches the generator and the sliders: `yawParam:'yaw'`, `pitchParam:'pitch'`, `rollParam:'roll'` (with `rollDefault:0`), identical in shape to the working `spiral3d` entry. Pitch range on the gizmo is unchanged (`0…89°`). (RGR: `tests/unit/renderer-3d-rotation-control.test.js` — dragging the mesh-topography yaw handle now updates `params.yaw` (not `rotate`) and leaves `rotate`/`tilt` undefined, and a third `rollHandle` exists and drives `params.roll`; both assertions fail against the old `rotate`/`tilt` spec. Full unit suite green — 1791 passing.)
- **3D parameter usability audit — Wave 2: polyhedron vertex "Hide Interior" now actually hides, Image Surface noise controls hide until you add a noise layer, and compass-heading dials replace two sliders (v1.1.116).** Three more audit findings, all geometry-neutral on default parameters. **(D1)** Polyhedron's `vertexOcclusionMode` "Hide Interior" (`occlude`) was inert — it only pushed a screen-space clip mask and never dropped a vertex glyph hidden behind a nearer face, so on the convex default buckyball it rendered identically to "Outline Only". It now drops a vertex marker entirely when its projected point falls inside a *nearer* opaque front face, reusing the existing depth-sorted `isPointOccludedByFaces` helper (front-faces only, strict nearer-depth test with a small epsilon, skipping the vertex's own incident faces so it can't occlude itself). Convex solids still show no change (no vertex is behind a nearer face there — correct), but self-occluding deformations (twist / shard / explode) and overlapping imported meshes now read as solid. Default mode is "Outline Only", so existing output and visual baselines are byte-identical. **(D3)** Image Surface's **Noise Mode** and **Noise Amount** controls rendered even with an empty noise stack, where they do nothing — `createNoiseField` returns null until an enabled noise layer exists. Both controls now carry a `showIf` that mirrors the generator's own enablement filter (`noises.some(n => n && n.enabled !== false)`), so they appear exactly when noise actually contributes; the always-visible Noise Stack list remains the place to add a layer. (An explicit "add a noise layer" empty-state hint is deferred — it needs a panel-renderer change.) **(UX7)** **Light Azimuth** (shared shading block) and **Plane Rotate** (Mesh Topography cutting plane) are compass headings that wrap a full circle, so they switch from a linear slider to the existing circular **angle dial** control type (as used by 3D Spiral's Start Longitude) — a better mental-model fit; values, ranges, and gating are unchanged. (RGR: `tests/unit/polyhedron-vertex-occlusion.test.js` — a pinned twisted+sharded icosahedron fixture drops strictly fewer vertex glyphs under `occlude` than `outline`, while `outline` and default output stay byte-identical; `tests/unit/3d-wave2-controls.test.js` — noiseMode/noiseAmount `showIf` is false for an empty/all-disabled stack and true once an enabled layer exists, and lightAzimuth/planeRotate are type `angle` while view yaw/pitch/roll stay sliders. Adversarially reviewed against all four generators; full unit + integration + visual suites green except two pre-existing, unrelated baseline failures.)
- **3D "Shading & Lines" controls now hide where they do nothing, and the duplicate-label confusion is gone — the first wave of the 3D-parameter usability audit (v1.1.115).** A harness-based audit of all four 3D algorithms (spiral3d, polyhedron, meshTopography, imageSurface) drove every parameter across its range and found a cluster of controls that render but are inert because the shared `SHADING_LINE_CONTROLS` block was spread verbatim into all four panels regardless of whether each algorithm's generator wires the underlying enhancement. The flat array is now a capability-driven factory `buildShadingControls(caps)` that ANDs each control's own self-toggle with a per-algorithm/per-mode capability predicate, so a control only appears where it actually affects output (single source of truth preserved — one factory, four call sites). Concretely: **spiral3d** (line/dot loops, no closed faces) hides Show Creases / Crease Angle / Lambert Hatching (+ light/hatch/cross-hatch) / Occlusion Bias — none have a surface to act on — while keeping the wired five (Depth Cue, Outline emphasis, Hidden Lines back-face/dash); **imageSurface** hides Outline, Creases, the **dead** Hidden Lines select (image-surface.js never reads `hiddenLineMode`; the existing *See-Through* control is the real toggle, so the injected select was both inert and a semantic duplicate) and Occlusion Bias, while gating Lambert Hatching to the mesh & bars modes where `pushFaceHatch` actually fires; **polyhedron** and **meshTopography** are unchanged (every shading control is wired — the factory falls through to the original self-toggles, byte-equivalent). Two label collisions are resolved: `focalLength` "Depth Strength" → **"Perspective Strength"**, `depthCueStrength` "Depth Strength" → **"Depth Cue Strength"** (they previously shared a visible label), `depthBias` "Depth Bias" → **"Occlusion Bias"**, and polyhedron's `faceOpacityMode` "Hidden Lines" → **"Hidden Faces"** (it collided with the shared `hiddenLineMode` "Hidden Lines"). The dead **Artwork Size** control was removed from Mesh Topography (its size is driven entirely by Scale X/Y/Z; `artworkSize` was an inert fallback that changed nothing — schema, default, and generator fallback all removed). All changes are geometry-neutral: hidden controls were no-ops on their algorithms and the relabels are display-only, so generated paths and every SVG visual baseline render byte-identical. (RGR: `tests/unit/3d-shading-capability.test.js` — spiral3d's nine face-only ids and imageSurface's outline/crease/hidden-line/occlusion-bias ids return `showIf:false` while the wired set returns true, hatch is gated to imageSurface mesh/bars, no two controls share a label within any 3D panel, and meshTopography no longer carries `artworkSize`; `tests/unit/mesh-topography-artworksize-removed.test.js` — sweeping `artworkSize` over its full range yields one identical geometry signature, proving it was inert. Adversarially reviewed against all four generator sources to confirm every *shown* shading control is wired and every *hidden* one is genuinely a no-op. Full unit + integration suites green.)
- **Image Surface's Curves toggle now takes effect when you flip it — the v1.1.111 fix was inert because the toggle never regenerated the geometry (v1.1.112).** v1.1.111 made the algorithm bake curves at *generate* time, but the generic controls-panel checkbox handler special-cases `curves` to only call `app.render()` (not `app.regen()`) — correct for 2D algorithms, where `curves` is a *draw-time* flag the renderer applies live, but inert for the 3D algorithms (spiral3d, polyhedron, meshTopography, imageSurface), whose paths are stamped `meta.straight` / `meta.forceCurves` at generate time. So toggling Curves on an Image Surface changed `params.curves` but never re-ran `generate()` → no visible change. And because the **Curve Smoothing** slider has `livePreview` (→ regen), moving it was the only thing that re-baked the geometry, making smoothing *look* like the on/off switch for curves (and, on the no-simplify topography back-compat path, look like a toggle with no gradient). The checkbox handler now regenerates instead of only rendering when the layer's algorithm bakes curves at generate time (`ALGO_DEFAULTS[type].is3d`), via a new `curvesBakedAtGenerate(layer)` guard — so the Curves toggle works live for all four 3D algorithms, and Curve Smoothing is a true gradient on top (measured: total bézier anchors fall 192→48 and mean handle length grows 1.28→7.36 as the slider runs 0→100). 2D algorithms keep the cheap render-only path. (RGR: `tests/integration/image-surface-curves-toggle.test.js` — clicking the real Curves checkbox on an Image Surface layer regenerates `forceCurves` béziers (and back to `straight` when unticked), where before the fix the paths stayed straight; a 2D `flowfield` layer's Curves toggle does **not** regen. Verified Red→Green.)
- **Image Surface's Curves toggle now actually curves the wires — turning Curves on bent nothing, in any mode (v1.1.111).** Every path Image Surface emits was stamped `meta.straight = true` at creation (`pathFromSurfaceSamples`/`pushVisibilityPaths`), and `meta.straight` unconditionally vetoes the layer **Curves** checkbox in both `Renderer.tracePath` and the SVG export tracer — so the toggle was inert for **Relief Lines** and **Deformed Mesh**, and **Topography** only ever curved because it ran its own `smoothToBezier` pass independent of the toggle. Image Surface now mirrors Mesh Topography's contract: **Curves** is the master enable, and the renamed **Curve Smoothing** slider (formerly *Contour Smoothing*, the same `contourSmoothing` param, now shown in any wire mode while Curves is on) drives bezier tension (with a floor so Curves-on stays smooth even at 0) plus a bounding-box-relative Visvalingam simplify tolerance (Curves-on trims the densest oversampling at ~0.4% of each path's diagonal; the slider drives it to ~2.4%), so the result is smooth **and** lean on export. Bars (2-point segments) and relief plane edges can't curve and are untouched. Back-compat is preserved exactly: Relief Lines / Deformed Mesh stay straight when Curves is off (they never curved before), and Topography keeps its legacy slider-only smoothing when Curves is off — so existing `.vectura` files and the `image-surface-topography` visual baseline render byte-identical. (RGR: `tests/unit/image-surface-curves.test.js` — Curves-off lines/mesh stay `meta.straight`; Curves-on turns every ≥3-point ribbon/row/contour into a `forceCurves` bezier with no `straight` flag; Curve Smoothing changes the geometry and a higher value yields a leaner anchor set; topography still smooths slider-only with Curves off; full unit + integration + visual suites green, baselines byte-identical.)
- **Image Surface "Image" noise layers now actually displace the surface — they were collapsing to a single corner pixel and rendering as a flat constant (v1.1.108).** Adding an `image`-type layer to an Image Surface's Noise Stack did nothing visible. `createNoiseField` routed every non-tiled, non-polygon layer through the generic `sampleScalar` FBM path, which samples in *world* space (`u · 1024`) scaled by the layer's tiny base zoom (`0.02`) — so the sample coordinate ran `0…~20` across the surface. `evaluate()`'s image branch clamps `u,v` to `[0,1]`, meaning ~97% of the surface read the raster's bottom-right corner pixel: a constant displacement that looked like the effect was off. Image layers now take a dedicated branch that maps the raster **1:1 across the whole surface in normalized [0,1] space**, single-octave (an image is not fractal noise, so FBM octaves would just stack clamped copies), honoring `angle`/`shift` for positioning. Tiled image layers (`tileMode ≠ off`) and all other noise types are unchanged. (RGR: `tests/unit/image-surface-noise-stack.test.js` — a left-black / right-white split image used as an image-noise layer now renders the left half of the height raster materially darker than the right half (it spanned the surface), where before the fix both halves read one identical constant; full unit + integration + visual suites green.)
- **Image Surface "Lines as Planes" now reads as a real solid surface — See-Through OFF actually hides what's behind nearer ribbons, and the planes no longer have a picket-fence of vertical dividers (v1.1.107).** Two bugs made the relief-ribbon (curtain) mode look wrong. First, each ribbon emitted a vertical drop segment at *every* interior sample, so the planes filled with a dense picket fence of verticals; a ribbon is a single curtain, so drops now belong only at the two end caps (left + right). Second, See-Through OFF did nothing but per-face normal culling — it never removed wires occluded by nearer ribbons, so the model always read as see-through. See-Through OFF now runs true inter-ribbon hidden-line removal: every curtain quad becomes a screen-space painter occluder (opaque from either side, tagged with its camera depth and a per-ribbon owner so a ribbon never occludes its own silhouette), and the wires behind a nearer crest are clipped — reusing the same `occludeBarEdges`/`Geometry3D.occludeSegments` pipeline the bars mode uses. See-Through ON is unchanged (full back lattice, dashed back edges). (RGR: `tests/unit/vectura-geometry-algorithms.test.js` — drop count is exactly `rows × 2` (no interior dividers), and a tall ridge band makes the solid pass carry materially fewer points than the see-through pass (occlusion is real); full unit + integration + visual suites green.)

### Added
- **Mesh Topography's view controller now matches 3D Spiral exactly — Yaw / Pitch / Roll (v1.1.113).** The v1.1.109 three-axis rotation shipped under the layer's legacy control names (**Rotate** / **Tilt** + a new Roll), which didn't match the 3D Spiral controller the request asked for. The mesh-topography **View** section now exposes the same **Yaw / Pitch / Roll** sliders as `spiral3d`, each spanning a full `−180…180°`. Under the hood the param ids were renamed `rotate → yaw` and `tilt → pitch` (roll unchanged); the engine reads only `yaw`/`pitch`/`roll`, and the one bundled preset (`torus-contours`) was migrated to the new keys. The default orientation is unchanged (`yaw −28`, `pitch 34`, `roll 0`), so visual baselines render byte-identical. The cutting-plane controls (`planeRotate`/`planeTilt`) are a separate, deliberately-distinct system and keep their names. **(v1.1.114: a transitional `rotate`/`tilt` read-fallback shipped with the rename was removed — pre-rename `.vectura` files are not supported, so the old keys are now ignored.)** (RGR: `tests/unit/mesh-topography-roll-axis.test.js` — each of yaw/pitch/roll changes the projection, an omitted roll is inert, and the dropped legacy `rotate`/`tilt` keys are ignored.)
- **Mesh Topography gains a full three-axis view rotation — Rotate / Tilt / Roll, each a full 360°, matching 3D Spiral (v1.1.109).** The mesh-topography viewport previously rotated on only two axes (`rotate` yaw + `tilt` pitch), and `tilt` was clamped to `0…89°` — you could never roll the model or tip it past the horizon. The viewport rotation now carries the third **Roll** (camera-Z twist) axis through both view-rotation call sites (`project3` projection + the contour face-culling pass), and **Tilt** is widened from `0…89°` to `−180…180°` so all three axes share the same full-360° span — the "same amount of rotation in each direction" parity with 3D Spiral's yaw/pitch/roll. `Geometry3D.rotatePoint` already supported the `roll` field, so this is a wiring + controls change with no rotation-math change. The cutting-plane rotation (`planeRotate`/`planeTilt`) is unchanged. Defaults are unchanged (`rotate −28`, `tilt 34`, new `roll 0`), so existing layers, presets, and visual baselines render byte-identical until Roll is moved or Tilt is pushed past 89°. (RGR: `tests/unit/mesh-topography-roll-axis.test.js` — changing `roll` materially changes the projected geometry where before it was a hard-coded no-op, an omitted `roll` matches `roll:0` for back-compat, and a fixed angle is deterministic; full unit + integration suites green — 1749 unit + 610 integration.)
- **3D Spiral gains two new wrap surfaces — Torus and Capsule (v1.1.106).** Following a product-lead assessment that rejected merging the 3D algorithms into one mega-renderer (the facet-based silhouette/crease/hatch enhancements don't apply to `spiral3d`'s smooth parametric surface, and merging layer-type strings would break `.vectura` presets/serialization keyed on `type`), the genuine low-cost win was simply *more surfaces under the existing spiral-wrap treatment*. `spiral3d`'s `shape` branch now adds **Torus** (`u` sweeps the major/toroidal ring once while `longitude` winds the minor/poloidal tube `turns` times, so a spiral coils around the donut; outline rings become the outer/inner equator since a torus has no poles) and **Capsule** (a cylinder of `capsuleHeight`/`capsuleRadius` capped by two hemispheres, with `u` allocated along the meridian arc length so the wrap pitch stays even across the cap↔barrel seams). Purely additive: no new layer type, no serialization change; depth-cue, see-through, dots, and emphasizeOutline all work unchanged. New params `torusRingRadius`/`torusTubeRadius`/`capsuleRadius`/`capsuleHeight` with `showIf`-gated panel controls; two presets (**Torus Coil**, **Capsule Wrap**). (RGR: `tests/unit/vectura-geometry-algorithms.test.js` — torus and capsule are finite, non-empty, and signature-distinct from the sphere and each other, and a degenerate `capsuleHeight: 0` capsule collapses byte-identically to a same-radius sphere (its meridian latitude reduces to the sphere's `(u−0.5)·π`); existing shapes and baselines unchanged. Full unit + integration + visual suites green — 2349 unit/integration + 21 visual.)
- **Version-stamped cache-busting so JS/CSS changes always take effect — no more stale-cache "my fix isn't showing up" (v1.1.105).** This is a no-build app: `index.html` lists ~150 local `<script>`/`<link>` tags with no bundler hash, so the browser caches every module by bare URL. Because `index.html` is always refetched (it's the document) while the modules are not, a soft reload could serve **stale JS** even though the in-app version badge (baked into `index.html`) looked current — the exact trap that hid the v1.1.103 Mesh Topography contour fix behind old cached `mesh-topography.js`. `npm run version:sync` now stamps `?v=<package version>` onto every local (`./`-prefixed) script and stylesheet (external CDN/font URLs left untouched; idempotent — replaces any existing `?v=…` rather than doubling it), and the runtime skin swap (`SkinManager.swapStylesheet`) appends the same query so switching skins can't load a stale palette either. The test runtime loader strips the query before resolving disk paths. (RGR: `tests/unit/index-html-cache-busting.test.js` — every local script + stylesheet carries `?v=<version>`, externals don't, no double-stamp; `skin-manager` href assertion updated to the new stamped contract. Full unit + integration + visual + e2e green — the app loads and runs with the query strings.)
- **Mesh Topography contours now actually respond to the Curves toggle and Contour Smoothing — and curve into optimized, elegant beziers instead of dead-flat or invisibly-smoothed polylines (v1.1.103).** Both controls previously looked inert: every sliced contour was stamped `meta.straight = true` at creation, so the layer **Curves** checkbox could never curve it (the `straight` flag always wins in `Renderer.tracePath` and the SVG export tracer), and **Contour Smoothing** *did* attach Catmull-Rom handles but the slice polylines are sampled at ~1–2px spacing, so the handles came out sub-pixel and the resulting cubic was visually identical to the chords — smoothing with no perceptible effect, and a "deconstructed" SVG carrying one cubic per sample (the opposite of elegant). Now `buildContours` treats **Curves** as the master enable (Curves on **or** Contour Smoothing > 0 ⇒ bezier; smoothing keeps implying curves for back-compat) and `Geometry3D.smoothToBezier` gained an opt-in `simplifyTolerance` that **Visvalingam-simplifies the polyline before fitting handles**. Tolerance scales with each contour's own bounding-box diagonal (Curves-on trims the densest oversampling at ~0.4% of the diagonal; the slider drives it to ~2.4%), with a tension floor so Curves-on stays smooth even at smoothing 0. Net effect on a default sphere: turning Curves on roughly halves the contour control-point count (1476 → 745) while curving every ≥3-point contour, and Contour Smoothing 80 cuts it ~75% (→ 368) — visibly smooth on canvas and lean, native-cubic curves on export. Defaults are unchanged (Curves off + Contour Smoothing 0 ⇒ verbatim straight polylines), so existing layers, presets, and visual baselines render byte-identical. `imageSurface` topography keeps its prior behavior (it doesn't pass the new option). (RGR: `tests/unit/vectura-geometry-algorithms.test.js` — Curves-on alone curves every curvable contour with no slider change; Contour Smoothing simplifies before fitting so the curved output has materially fewer control points than the raw samples and more smoothing yields a sparser curve; existing smoothing/no-op contracts retained. Full unit + integration + visual suites green; baselines byte-identical.)
- **The 3D suite gains four shared, plotter-native rendering enhancements — depth cueing, silhouette/feature-edge weight, true hidden-line removal, and Lambert hatching (v1.1.102).** Built once in `src/core/algorithms/geometry3d.js` and wired into all four 3D algorithms (`meshTopography`, `polyhedron`, `spiral3d`, `imageSurface`); every feature defaults **off**, so existing output and all visual baselines stay byte-identical until a toggle is flipped.
  - **#2 Depth cueing** (`applyDepthCue`): each path is tagged with its camera-space depth and, when enabled, gets a `meta.strokeDash` scaled by distance (near = solid/tight, far = sparse) — atmospheric perspective through the already-honored dash channel, with hidden-line dashes taking precedence. Controls: `depthCue` (Off | Dash by depth) + `depthCueStrength`. Applies to all four algos.
  - **#3 Silhouette & feature-edge emphasis** (`extractSilhouette`/`extractCreases` + a new `meta.weightScale` stroke-width channel honored in the canvas renderer and both SVG export paths): a confident heavy profile + crease lines over thin interior wireframe. Controls: `emphasizeOutline` + `outlineWeight`, `showCreases` + `creaseAngle`. (meshTopography & polyhedron; spiral3d weights its outline rings.)
  - **#4 True hidden-line removal** (`occludeSegments`, screen-space painter occlusion with per-segment AABB culling + depth bias): a near face now actually hides the edges/contours behind it, instead of only back-face culling — the defining quality of CAD pen drawings. `hiddenLineMode` (Back-face only | Remove hidden | Dash hidden) + `depthBias`. (meshTopography & polyhedron; spiral3d & imageSurface use their existing surface-normal visibility.)
  - **#5 Illumination-driven Lambert hatching** (`hatchPolygon`/`lambertHatch` + a shared `resolveLight` azimuth/elevation vector): front faces fill with parallel scan-line strokes whose spacing follows `N·L` (denser = darker), with an optional cross-hatch pass — tonal shading rendered entirely as plotter strokes. `hatchEnable`, `lightAzimuth`/`lightElevation`, `hatchAngle`, `hatchSpacing`, `crossHatch`. (meshTopography, polyhedron, and imageSurface mesh/bars; spiral3d/topography no-op cleanly — no facets to clip.) Spacing floors + `fastPreview` down-rating bound the stroke count.
  - Built with a parallel **foundation team** (worktree) + four parallel **per-algorithm implement→adversarial-review→judge teams** (one worktree each, disjoint files), integrated and merged back to main. RGR: `tests/unit/geometry3d-enhancements.test.js` (24 — pure-helper coverage: hatch clipping, occlusion hide/keep, depth-cue dash + hiddenLine precedence, silhouette edges, light vector) + per-algorithm toggle tests in `tests/unit/vectura-geometry-algorithms.test.js`; full unit + visual suites green (1755) with baselines byte-identical (all toggles default off).
- **Image Surface gains the universal Noise Rack — a live, multi-layer noise stack that displaces the 3D surface (v1.1.99).** The `imageSurface` algorithm now samples `params.noises` through `window.Vectura.NoiseRack` (the same stack topo/flowfield use, per the Arch-7 convergence contract) and folds the result into its height field, so every relief line, mesh vertex, contour, and bar physically rides the noise — visible 3D noise, not a flat texture. - **The 3D algorithm suite gets STL import, five new mesh primitives, true bezier smoothing, perspective projection, detail to 100, and a tunable move-preview (v1.1.98).** A sweep across the four 3D-capable algorithms (`meshTopography`, `polyhedron`, `spiral3d`, `imageSurface`) sharing `src/core/algorithms/geometry3d.js`:
  - **STL import.** New `src/core/stl-parser.js` parses binary *and* ASCII `.stl` into a welded, normalized (`[-1,1]` centred), JSON-serialisable `{vertices,faces}` mesh (large meshes uniformly down-sampled to ≤12k faces with vertex re-indexing so they stay light enough for `layer.params`/undo/`.vectura`). `meshTopography` gains a **STL Mesh** source and `polyhedron` a **STL Mesh** solid — the imported mesh flows through the existing wireframe / depth-plane-contour / silhouette and face/edge/vertex machinery (hidden lines, smoothing, perspective all apply). Input handling copies into a realm-local buffer so `FileReader`/test-sandbox `ArrayBuffer`s are safe.
  - **Five new mesh-topography primitives:** Cylinder, Capsule (hemispherical caps), Pyramid (detail-subdivided square taper), Superellipsoid (rounded-cuboid superquadric), and Torus Knot (swept (2,3)-knot tube) — each geometrically distinct from the sphere fallback.
  - **Smoothing now makes true curves.** The 3D contour/line smoothers no longer corner-cut into dense straight polylines; `Geometry3D.smoothToBezier` attaches Catmull-Rom cubic handles (`meta.anchors` + `forceCurves`, dropping `meta.straight`) so a `*Smoothing` slider curves the line on screen **and** in export. `ui.pathToSvg` and the export-modal preview tracer gained a native cubic `C` branch mirroring `Renderer.tracePath`, so plot output matches the canvas exactly (this also fixes the morph modifier's smooth rings exporting as faceted polylines). Smoothing at 0 is a byte-identical no-op; hidden-line dashes survive smoothing.
  - **Perspective projection.** `Geometry3D.projectPoint` gained an optional one-point pinhole divide (focal + camera distance, near-plane clamped) selected by a shared **Projection** control on all four algorithms; **Orthographic is the default and byte-identical** to the prior arithmetic (back-face/normal visibility, computed pre-projection, is unaffected).
  - **Primitive detail to 100** (`meshTopography`, was capped at 48).
  - **Higher-fidelity 3D move preview + a Document Setup control.** Live-drag regen fidelity is now driven by `Geometry3D.previewDetailScale`/`previewCap` and a new **3D move preview** setting (Draft / Balanced / High) in Document Setup → Guides & Display; the default (Balanced) is already crisper than the old hardcoded factors, and full renders never down-rate. The setting persists in cookies and `.vectura` (validated enum). ### Performance
- **Petalis no longer bogs the browser down when you raise the petal count — generation is now ~10–12× faster with byte-identical output (v1.1.93).** Changing petal count on a Petalis layer ran the per-petal occlusion clip (`clipSegmentOutside`) and the per-shading-line inside-the-outline clip (`clipSegmentInsidePolygon`) by testing **every segment against every edge of every overlapping occluder/polygon**, with the polygon's bbox + edge list rebuilt on each segment — an O(n²)-in-petal-count blowup dominated by ~3 million `segmentIntersection` calls at 96 petals (profiled). - **Morph Modifier Group — graduated shape-blending between 2+ child layers, integrated into the modifier-container system (v1.1.65).** A second modifier type alongside Mirror. Drop two or more layers into a Morph group and it fills the space between their shapes with N interpolated in-between rings — a circle becoming a wavetable, Rings blooming into a Lissajous, a Harmonograph→Wavetable→Spiral transition strip. The mental model is the Illustrator/Inkscape Blend tool, but pipeline-integrated and plotter-native: every emitted path is a polyline.
